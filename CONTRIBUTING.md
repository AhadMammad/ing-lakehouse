# Contributing to ing-lakehouse

## First-time setup

```bash
cp .env.example .env       # then edit secrets (RUSTFS_*, JUPYTER_TOKEN, AIRFLOW_ADMIN_PASSWORD, POSTGRES_SOURCE_PASSWORD)
make up                    # start the local stack
```

`.env` is gitignored. Per-instance overrides live in `.env.<whoami>` (also gitignored, generated by `make init-instance`).

## Branching

- Work on short-lived branches off `main`: `feat-*`, `fix-*`, `chore-*`, `docs-*`.
- Open a Pull Request against `main`.
- PRs are squash-merged. The PR title becomes the single commit on `main`, so **your PR title must follow Conventional Commits** (see below).

## Conventional Commits

The PR title is what ends up in the Git history and drives automated versioning. It must match:

```text
<type>(<scope>)!?: <subject>
```

### Allowed types

| Type       | Bumps version? | Use for                                                       |
|------------|----------------|---------------------------------------------------------------|
| `feat`     | minor          | A new user-visible capability                                 |
| `fix`      | patch          | A bug fix                                                     |
| `perf`     | patch          | Performance improvement                                       |
| `refactor` | patch          | Code change that neither fixes a bug nor adds a feature       |
| `docs`     | none           | Documentation only                                            |
| `test`     | none           | Adding or fixing tests                                        |
| `build`    | none           | Build system / Dockerfile / dependency changes                |
| `ci`       | none           | CI/CD configuration changes                                   |
| `chore`    | none           | Maintenance work that doesn't fit elsewhere                   |
| `revert`   | patch          | Reverts a previous commit                                     |

### Allowed scopes

`data-generator`, `etl-app`, `airflow`, `trino`, `iceberg`, `nessie`, `rustfs`, `kafka`, `jupyter`, `infra`, `docs`, `ci`, `deps`.

Scope determines **which package** gets bumped:

- `data-generator` → bumps `services/data-generator`
- `etl-app` → bumps `services/etl-app`
- Anything else (or no scope) → bumps the root `ing-lakehouse` platform version

### Breaking changes

Append `!` after the type/scope **or** add a `BREAKING CHANGE:` footer in the PR body. Either triggers a **major** version bump.

### Examples

```text
feat(etl-app): add bronze partition pruning
fix(data-generator): respect rows_per_day cap
chore(deps): bump pyiceberg to 0.11.2
docs: explain medallion layout in README
feat(airflow): add SLA miss callback to seeder DAG
feat!: switch catalog from Nessie to Polaris
```

## Release process

You don't cut releases manually. After your PR merges to `main`:

1. The `release-please` workflow opens (or updates) a **Release PR** for any package whose pending commits warrant a release.
2. Reviewing & merging the Release PR cuts the Git tag and publishes a GitHub Release with auto-generated notes.
3. Tag formats:
   - Platform: `vX.Y.Z`
   - Services: `data-generator-vX.Y.Z`, `etl-app-vX.Y.Z`

## Local checks before opening a PR

```bash
# validate compose files
docker compose -f docker-compose.yml config -q
docker compose -f docker-compose.local.yml config -q

# lint Python services
uvx ruff check services/data-generator services/etl-app
```
