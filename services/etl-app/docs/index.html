etl-app

A lightweight Python container that runs the lakehouse's medallion ETL: three independent pipelines (crypto, weather, payments) materializing bronze / silver / gold Apache Iceberg tables on RustFS, cataloged by Nessie, spawned on-demand by Airflow's DockerOperator.

1. Overview

etl-app is a single Python package (etl_app/) packaged into a Docker image. It is not a long-running service — Compose builds it under the build-only profile (services/etl-app/docker-compose.yml), and Airflow's DAG layer spawns ephemeral containers for each task. The image entrypoint is python -m, so DAGs pass module paths as commands: 

docker run --rm \
  -e NESSIE_URI=http://nessie:19120/iceberg \
  -e AWS_S3_ENDPOINT=http://rustfs:9000 \
  -e AWS_ACCESS_KEY_ID=… -e AWS_SECRET_ACCESS_KEY=… \
  ing-lakehouse-etl-app:latest \
  etl_app.jobs.crypto_bronze --date 2026-05-21

Design principles

  • Medallion — bronze (raw, append-only), silver (typed + deduped), gold (business-ready aggregates).
  • Idempotent by logical date — re-running a job for the same --date produces the same table state. Bronze snapshots overwrite per logical_date; silver upserts on natural keys; gold marts overwrite per partition (or full-table for CLV).
  • No framework — module-level constants instead of a Settings class, argparse CLIs, polars for in-memory transforms, pyiceberg for table I/O.
  • One image, many jobs — same container runs every job; the command line decides which.

2. Architecture

The etl-app sits between source systems and the lakehouse query layer:

┌──────────────────────────┐ ┌──────────────────────────┐ │ External APIs │ │ postgres-source │ │ • CoinGecko /markets │ │ payments OLTP (7 tables)│ │ • Open-Meteo /forecast │ │ seeded by data-generator│ └─────────────┬────────────┘ └─────────────┬────────────┘ │ │ │ requests / connectorx │ ▼ ▼ ┌─────────────────────────────────────────────┐ │ etl-app (Docker) │ │ ┌─────────────────────────────────────┐ │ │ │ jobs/crypto_*.py jobs/weather_*.py│ │ │ │ jobs/payments_*.py │ │ │ │ catalog.py · _iceberg_utils.py │ │ │ └────────────┬────────────────────────┘ │ └────────────────┼────────────────────────────┘ │ PyIceberg over REST ▼ ┌─────────────────────────────────────────────┐ │ Nessie REST Catalog (port 19120) │ ← namespaces + table metadata └─────────────────────────────────────────────┘ │ S3 (path-style) ▼ ┌─────────────────────────────────────────────┐ │ RustFS (S3-compatible) │ │ bucket: iceberg-warehouse │ │ └── warehouse/{bronze,silver,gold}/… │ └─────────────────────────────────────────────┘ │ read by ▼ Trino · Spark · Jupyter (PyIceberg)

Inside the etl-app, every job follows the same shape: extract → transform (polars) → write (PyIceberg). The catalog.py module is the single point of contact with Nessie; the load_or_create helper makes table creation race-safe so parallel TaskGroups don't collide on first run.

4.3 Payments — Postgres OLTP → Kimball star → marts

The largest pipeline. Reads from postgres-source (seeded by data-generator) into 7 bronze tables, then a Kimball star (4 dims + 4 facts) in silver, then 4 business marts in gold.

Bronze — 7 tables

TableSourceModeFilter
bronze.payments_merchants_rawmerchantssnapshotfull table
bronze.payments_customers_rawcustomerssnapshotfull table
bronze.payments_methods_rawpayment_methodssnapshotfull table
bronze.payments_payments_rawpaymentsincrementalWHERE updated_at::date = '{date}'
bronze.payments_refunds_rawrefundsincrementalWHERE updated_at::date = '{date}'
bronze.payments_fees_rawfeesincrementalWHERE created_at::date = '{date}'
bronze.payments_settlements_rawsettlementsincrementalWHERE settlement_date = '{date}'

All bronze tables append the housekeeping columns ingest_ts (timestamptz, required), logical_date (date), source (literal postgres-source), and raw_payload (JSON of source columns). UUIDs are cast to text in the SQL template so connectorx → Arrow conversion stays clean. Money columns use DecimalType(14, 2).

Silver — Kimball star (4 dims + 4 facts)

TableKindNatural keyPartitionSource bronze
silver.dim_customerdim (SCD-1)customer_id—customers
silver.dim_merchantdim (SCD-1)merchant_id—merchants
silver.dim_payment_methoddim (SCD-1)method_id—methods
silver.dim_datedim (generated)date—none (synthesized from --date)
silver.fact_paymentsfactpayment_ididentity(payment_date)payments
silver.fact_refundsfactrefund_ididentity(refund_date)refunds
silver.fact_feesfactfee_ididentity(fee_date)fees
silver.fact_settlementsfactsettlement_ididentity(settlement_date)settlements

Silver transforms (see payments_silver.py):

  • dim_customer — derives age_band from birthdate; updated_at bumped on every refresh.
  • dim_payment_method — derives is_expired from (expiry_year, expiry_month).
  • dim_date — emits one row for --date with day, month, quarter, year, day_of_week, iso_week, is_weekend.
  • fact_payments — dedup on payment_id (keep latest ingest_ts), derive payment_date from created_at, compute settled_lag_hours = (settled_at − created_at) / 3600.
  • fact_refunds / fact_fees — same dedup + derived date pattern.
  • fact_settlements — uses settlement_date directly.

Gold — 4 business marts

MartGrainPartitionRefreshReads
gold.daily_payment_summaryone row per dayidentity(day)overwrite by dayfact_payments, fact_refunds, fact_fees
gold.daily_revenue_by_merchant(merchant, day)identity(day)overwrite by dayfact_payments, fact_refunds, fact_fees, dim_merchant
gold.customer_lifetime_valueone row per customer—full-table recomputefact_payments, fact_refunds, dim_customer
gold.merchant_settlement_daily(merchant, settlement_date)identity(settlement_date)overwrite by settlement_datefact_settlements, fact_payments, dim_merchant

Mart highlights (see payments_gold.py):

  • daily_payment_summary: total_payments, total_volume, success/failed counts, success_rate_pct, avg_amount, unique customers/merchants, total refunds, total fees, net_revenue.
  • daily_revenue_by_merchant: gross / net volume per merchant, txn count, success count, joined with merchant name + category.
  • customer_lifetime_value: total spend, total / successful / failed payment counts, total refunds, refund_rate_pct, first_payment_date, last_payment_date, active_days.
  • merchant_settlement_daily: settlement amount + count joined with same-date payments to compute settlement_lag_days and gross-volume reconciliation.

6. Orchestration

Every etl-app job is invoked as a DockerOperator task in an Airflow DAG. The Airflow worker mounts the host Docker socket and spawns a fresh container per task using ETL_APP_IMAGE and LAKEHOUSE_NETWORK env vars (set by Compose at worker startup).

DAGs that run etl-app

DAGScheduleTasksFile
crypto_etl_top20 0 2 * * * (02:00 UTC) bronze → silver → gold (sequential) crypto_etl.py
weather_etl_baku 0 6 * * * (06:00 UTC) bronze → silver → gold (sequential) weather_etl.py
payments_etl 0 3 * * * (03:00 UTC) bronze (7 parallel) → silver.dims (4 parallel) → silver.facts (4 parallel) → gold (4 parallel) payments_etl.py
payments_source_seeder manual trigger runs data-generator image (not etl-app) to populate postgres-source payments_seeder.py

7. Local Development

Build the image

make build-etl-app
# → docker compose --profile build-only build etl-app
# → tags <project>-etl-app:latest

Tail build logs

make logs-etl-app

Run a job manually

docker run --rm \
  --network ing-lakehouse_lakehouse-net \
  -e NESSIE_URI=http://nessie:19120/iceberg \
  -e AWS_S3_ENDPOINT=http://rustfs:9000 \
  -e AWS_ACCESS_KEY_ID=rustfsadmin \
  -e AWS_SECRET_ACCESS_KEY=rustfsadmin123 \
  -e ICEBERG_WAREHOUSE_BUCKET=iceberg-warehouse \
  ing-lakehouse-etl-app:latest \
  etl_app.jobs.weather_bronze --date 2026-05-21

Seed payments data

make run-data-generator START=2026-01-01 END=2026-05-27 ROWS=500
# or trigger the payments_source_seeder DAG with conf:
# { "start_date": "2026-01-01", "end_date": "2026-05-27", "rows_per_day": 500 }

Inspect tables from Jupyter

from etl_app.catalog import get_catalog
catalog = get_catalog()
catalog.list_namespaces()                           # [('bronze',), ('silver',), ('gold',)]
catalog.load_table(("silver", "fact_payments")).scan().to_arrow()

Idempotency: any job can be safely re-run for the same --date. Bronze incremental tables overwrite their date slice, silver upserts on natural keys, gold marts overwrite their partition. The one exception is bronze append-only tables (crypto and weather) — re-running them produces duplicate rows that silver dedups out.

Exit semantics

  • Exit 0 on success (including empty result — empty bronze pull is a no-op, not a failure).
  • Exit non-zero on uncaught exception. Airflow surfaces the stderr in the task log.
  • Argparse validation errors exit 2 (standard Python behavior).
